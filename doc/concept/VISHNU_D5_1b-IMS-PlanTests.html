        Présentation du document

        Objectifs du document

Les objectifs de ce document sont les suivants :

  1. Présenter les fonctionnalités du module IMS à tester.

  2. Présenter les différents types de tests  permettant  de valider les cas d'utilisation de chaque fonctionnalité du module IMS.

  3. Présenter les méthodes et outils utilisés pour ces tests.

  4. Donner des détails de l'environnement de test

        Structure du document

Le chapitre 1 présente le document.

Le chapitre 2 présente les environnements et outils de tests.

La chapitre 3 décrit la liste des tests qui s'appliquent aux fonctionnalités à tester.

La chapitre 4 présente les outils utilisés pour ces tests.

  Environnements et outils de tests

 Environnements de tests

Pour ces tests, nous allons utiliser l'environnement de tests suivants avec des systèmes dédiés aux tests:

  1. Système d'exploitation : c'est le système Linux décrit dans [STB] qui sera installé sur toutes les machines clientes et serveurs.

  2. Nombre de machines clientes : une machine cliente décrite dans [STB] pour pouvoir lancer des tests à distance (selon le cas de test).

  3.  Nombre de machines serveurs : deux serveurs IMS décrit dans [ARCH] .

  4. Un serveur SysFera-DS dédié décrit dans [ARCH].

  5. Un serveur UMS décrit dans [ARCH].

  6. Un serveur TMS et FMS décrit dans [ARCH]

  7. Le module LogService de SysFera-DS décrit dans [STB].

       

 Outils de tests

Les outils suivants seront utilisés pour les tests :

  1. Hudson (serveur d'intégration)  sera utilisé pour les tests d'intégration continue (compilation, installation).

  2. La librairie Boost test sera utilisée pour la réalisation des tests unitaires en C++.

  3. Le framework JUnit sera utilisé pour  la réalisation des tests unitaires en JAVA.

  4. Des scripts Python seront utilisés pour la réalisation des tests unitaires en Python.

  5. Des scripts SHELL BASH seront utilisés pour la réalisation des tests unitaires de la ligne de commande.

  6. Squore et Logiscope seront utilisés pour la qualité du code.

  7. Valgrind sera utilisé pour la détection de fuite mémoire.

        Tests fonctionnels

Les tests fonctionnels ou tests de boites noires permettent à partir des données en entrée d'observer les réactions du logiciel à tester.

Les fonctionnalités sont classées par catégories de tests :

  1. Catégorie 1 : comprend la fonctionnalité permettant d'exporter toutes les commandes ayant été exécutées durant une session: exportCommands.

  2. Catégorie 2 : regroupe les fonctionnalités permettant d'agir physiquement sur un composant :

    loadShed, stop, restart.

  3. Catégorie 3 : comprend le service permettant de consulter l'ensemble des processus VISHNU : getProcesses.

  4. Catégorie 4 :  regroupe les fonctionnalités permettant la consultation et modification des informations système : getMetricCurrentValue, getMetricHistory, getSystemThreshold, setSystemThreshold, getUpdateFrequency, setUpdateFrequency, defineUserIdentifier, defineMachineIdentifier, setSystemInfo,  getSystemInfo, defineJobIdentifier, defineTransferIdentifier.

Chaque test fonctionnel sera réalisé en utilisant chacune des interfaces d'accès suivantes : fonctions API/C++ et API/Python, ligne de commande utilisateur et API web services.

Nous allons dans cette section présenter les différentes stratégies de tests en utilisant des programmes exécutables ou des fonctions de l'API IMS puis en observant les résultats obtenus. Les deux conditions à vérifier sont :

  1. Est ce que le programme réalise les fonctions attendues en accord avec les spécifications?

  2. Est ce que le programme fonctionne correctement ?

 Identifiant de tests

Les identifiants des tests sont de la forme: <ID DSG>-[B|E]:<NOM_TEST>-<INTERFACE> où 'ID DSG' est l'identifiant du cas d'utilisation dans le document [SDG] , 'B' la séquence de base (exécution du cas d'utilisation avec des paramètres valides), 'E' la séquence d'exception (comportement du cas d'utilisation en cas d'erreurs),  'NOM_TEST' le nom du test décrit dans [SDG] et 'INTERFACE' les différents codes des interfaces utilisées par VISHNU : fonctions API/C++(CPP), API/Python (PY), ligne de commande utilisateur (CLI) et API web services  (WS).  L'absence du code de l'interface signifie que le test s'applique à toutes les interfaces précédentes.

Il faut noter que les tests avec des paramètres corrects sont effectués dans une session active avec un mode de fermeture à la déconnexion (CLOSE_ON_DISCONNECT) et un certificat de session valide.

 Tests de stress       

 Présentation

Ces tests permettent de vérifier que la cible de test peut réussir à basculer et à récupérer pour différents dysfonctionnements matériels, logiciels ou réseaux sans qu'il y ait perte de données. Ils vérifient que lorsqu'une défaillance survient un système alternatif ou un système de sauvegarde prends le relai correctement sans perte de données ou de transactions.

 Environnement des tests

Nous utiliserons l'environnement de test 1.

 Identifiants de tests

Les identifiants des tests sont de la forme: STR-<Element stressé>-<Type de stress>.
