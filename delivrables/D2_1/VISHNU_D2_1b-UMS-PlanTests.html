        Présentation du document

        Objectifs du document

Les objectifs de ce document sont les suivants :

  1. Présenter les fonctionnalités du module UMS à tester.

  2. Présenter les différents types de tests  permettant  de valider les cas d'utilisation de chaque fonctionnalité du module UMS.

  3. Présenter les méthodes et outils utilisés pour ces tests.

  4. Donner des détails de l'environnement de test

        Structure du document

Le chapitre 1 présente le document.

Le chapitre 2 présente les environnements et outils de tests.

La chapitre 3 décrit la liste des tests qui s'appliquent aux fonctionnalités à tester.

La chapitre 4 présente les outils utilisés pour ces tests.

        Glossaire, abréviations et acronymes

        Glossaire

  1. Script de test : c'est un programme qui contient les instructions permettant de réaliser les tests.

  2. Certificat : correspond à la clé de session (identifiant de session crypté)

        Acronymes

  1. STB : Spécification Technique des Besoins

  2. DSG : Document de Spécification générale

  3. DSD : Document de Spécification Détaillées

  4. API : Application Programming Interface (Interface de programmation d'applications)

  5. MPI : Message Passing Interface

     

        Abréviations

  1. Id.         Identifiant

  2. Réf.        Référence

  3. Rel.#        Numéro de release/version

  4. Resp.        Responsable(s)

  Environnements et outils de tests

 Environnements de tests

Pour ces tests nous allons utiliser les deux environnements  tests suivants :

 Environnement test 1:

  1. Système d'exploitation : c'est le système linux qui sera installé sur toutes les machines clientes et serveurs.

  2. Nombre de machines clientes : au moins 1 machine cliente décrite dans [STB] pour pouvoir lancer des tests à distance (selon le cas de test).

  3. Deux serveurs UMS décrit dans [STB].

  4. Un serveur SysFera-DS dédié décrit dans [STB].

  5. Un serveur de base de donnée Oracle 11.2.0.2.

  6. Un serveur web JBOSS décrit dans [STB].

Environnement test 2:

  1. Système d'exploitation : c'est le système linux qui sera installé sur toutes les machines clientes et serveurs.

  2. Nombre de machines clientes : au moins 1 machine cliente décrite dans [STB] pour pouvoir lancer des tests à distance (selon le cas de test).

  3. Deux serveurs UMS décrit dans [STB].

  4. Un serveur SysFera-DS dédié décrit dans [STB].

  5. Un serveur de base de donnée PostgreSQL.

  6. Un serveur web JBOSS décrit dans [STB].

Pour chacun des environnements, nous allons utiliser une base de données avec 100 utilisateurs, 10 machines serveurs,

10 000 sessions et 100 000 enregistrements de commandes utilisateurs.

 Outils de tests

Les outils suivants seront utilisés pour les tests :

  1. Hudson (serveur d'intégration)  sera utilisé pour les tests d'intégration continue (compilation, installation).

  2. Boost test sera utilisé pour la réalisation des tests unitaires.

  3. Des scripts batchs seront utilisés pour la réalisation des scripts de tests, des tests fonctionnels.

  4. Squore et Logiscope seront utilisés régulièrement pour la qualité du code.

        Tests fonctionnels

Les tests fonctionnels ou tests de boites noires permettent à partir des données en entrée d'observer les réactions du logiciel à tester.

Les fonctionnalités (fonctions API/C++ et API/Python, ligne de commande utilisateur, API web service) sont classées par catégories de tests. A chaque catégorie correspond un ensemble de fonctionnalités. Les catégories de tests sont les suivantes :

  1. Catégorie 1 : regroupe les fonctionnalités permettant la gestion des sessions qui sont : connect, reconnect, listSessions, listHistoryCmd et close.

  2. Catégorie 2 : regroupe les fonctionnalités permettant la gestion des utilisateurs qui sont : addVishnuUser, updateUser, deleteUser, listUsers, changePassword et resetPassword.

  3. Catégorie 3 : regroupe les fonctionnalités permettant la gestion des configurations qui sont : addLocalAccount, updateLocalAccount, deleteLocalAccount, ListLocalAccount, saveConfiguration, restoreConfiguration,  configureDefaultOption et listOptions.

  4. Catégorie 4 :  regroupe les fonctionnalités permettant la gestion des machines qui sont : addMachine, updateMachine, deleteMachine et listMachines.

  Nous allons dans cette section présenter les différentes stratégies de tests en utilisant des programmes exécutables ou des fonctions de l'API UMS puis en observant les résultats obtenus. Les deux conditions à vérifier sont :

  1. Est ce que le programme réalise les fonctions attendues en accord avec les spécifications?

  2. Est ce que le programme fonctionne correctement ?

 Identifiant de tests

Les identifiants des tests sont de la forme: <ID DSG>-[B|E] où 'ID DSG' est l'identifiant du cas d'utilisation dans le document [SDG] , 'B' la séquence de base (exécution du cas d'utilisation avec des paramètres valides) et 'E' la séquence d'exécution (comportement du cas d'utilisation en cas d'erreurs)

U1.1-B1:connect

  1. Cas de test : les paramètres sont corrects et enregistrés dans la base de données.

  2. Vérifier qu'il n'y a aucune session ouverte sur la machine de test. Sinon désactiver la session ouverte.

  3. Lancer le programme connect avec comme paramètres le userId (login) et le mot de passe de l'utilisateur.

  4. Un certificat de session est créé et envoyé à l'utilisateur.

  5. Lancer  la commande listSessions avec la valeur du certificat retourné. Si le certificat est valide, l'exécution de listSessions réussit donc la connexion est validée.

La durée d'inactivité de la session doit avoir une valeur assez élevée pour permettre l'exécution du test dans de bonnes conditions

U1.1-E1:connect U1.1-E2:connect

  2. Vérifier qu'il n'y a aucune session ouverte sur la machine de test. Sinon désactiver la session ouverte.

  3. Lancer le programme connect avec le cas de test correspondant (E1 ou E2)

  4. Lancer une commande (listSessions) et vérifier que son exécution échoue.

U1-B2:close

  1. Cas de test : les paramètres sont corrects et enregistrés dans la base de données.

  1. Vérifier qu'il n'y a aucune session ouverte sur la machine de test. Sinon désactiver la session ouverte.

  2. Lancer le programme connect avec comme paramètres le login et le mot de passe de l'utilisateur et l'option choisie.

  3. Un certificat de session est créé et envoyé à l'utilisateur.

  4. Lancer  la commande listSessions avec la valeur du certificat retourné. Si le certificat est valide, l'exécution de listSessions réussit (Cici permet de vérifier que la session a été correctement ouverte).

Pour le cas B1 :

  1. Fermer le poste  ou le terminal sur lequel le programme connect a été lancé.

  2. Démarrer le poste client ou le terminal et vérifier que l'exécution d'un programme échoue (ex : listSessions) .

Pour le cas B2 :

  1. Ne lancer aucun programme pendant un certain temps (temps supérieur au temps d'inactivité fixé).

  2. Lancer la commande listSessions et vérifier que son exécution échoue.

Cas B2 :

- La session a été fermée automatiquement après le temps d'inactivité fixé

U1.5-B:reconnect

  1. Cas de test : paramètres valides.

  2. Lancer  le programme connect  dans deux environnements capables de se connecter à VISHNU pour avoir deux  identifiants de session dans la base de données par le même utilisateur.

  3. Faire un  listSessions dans chacun des environnements et récupérer l'identifiant de session.

  4. Essayer de se reconnecter en utilisant l'identifiant de session de l'autre environnement.

  5. Vérifier que la reconnexion s'est bien passée dans les deux environnements.

- Le certificat courant a été réinitialisé au certificat correspondant à l'identifiant de session et le login donné en paramètre.

  3. Son exécution échoue

U1.3.5-B:listSessions

  1. Cas de test : connexion en tant qu'utilisateur VISHNU

  2. Ouvrir plusieurs sessions en lançant les commandes connect l'une après l'autre.

  3. Lancer  la commande  listSessions sans paramètre.

  4. Inspecter la base de données pour vérifier que le nombre de sessions retournées par la commande listSessions est égal au nombre de sessions dans la base de données, que  chaque identifiant est bien enregistré dans la base de données.

  5. Désactiver les sessions ouvertes en utilisant plusieurs fois la commande close

- La liste des sessions de l'utilisateur est affichée.

  3. Son exécution échoue
